# 如何写好周志 / 月志 / 项目复盘

> 来源：问问大象复盘教学

## 零、核心框架：复盘六问

所有复盘，无论是周复盘、月复盘还是项目复盘，都以这六问作为外层骨架：

1. **回顾目标**：我这个周期或这个项目原本想达成什么？现在回头看，目标合理吗？
2. **评估结果**：结果是什么？达成度如何？和预期偏差在哪里？
3. **分析原因（正向）**：有哪些地方超预期？为什么会超预期？
4. **分析原因（负向）**：有哪些地方低于预期？根因是什么？
5. **重来演练**：如果再来一次，我会怎样做？
6. **后续规划**：接下来目标、路径、机制要怎么调整？

## 一、统一输出协议

从现在开始，**周复盘 / 月复盘 / 项目复盘**统一遵守以下协议：

- 六问必须保留为显式一级标题
- 新增的主题综合、多视角证据、假说追踪、边界说明，只能扩展到六问内部
- 不再允许用“执行摘要”“主题一/二/三”“咨询讨论”等新一级标题替代六问

统一一级标题如下：

1. `## 一、回顾目标`
2. `## 二、评估结果`
3. `## 三、分析原因（正向）`
4. `## 四、分析原因（负向）`
5. `## 五、重来演练`
6. `## 六、后续规划`

总原则：

**标题守旧，内容升级。**

## 二、为什么这样设计

这个统一协议要同时满足四件事：

1. 用户每次打开复盘，都能快速定位到熟悉的位置。
2. 系统保留主题归并、交叉验证、证据边界这些优化成果。
3. 周复盘、月复盘、项目复盘看起来是一套体系，而不是三套互不兼容的模板。
4. 以后新增功能时，不需要推翻模板，只需要在六问内部加插槽。

所以，六问负责稳定导航；主题综合、多视角分析、假说验证，负责提升六问内部质量。

## 三、六问内部如何扩展

### 0. 方向锚点与缺席项检查

周复盘和月复盘不只检查“发生了什么”，也检查“重要但缺席了什么”。

如果用户存在阶段性主线，例如考公、行测、申论、作息稳定、求职、项目交付，复盘必须先确认这些方向锚点是否在本周期出现行动证据。若没有出现，不应直接视为“不重要”，而应区分：

- `有推进`：有明确行动证据。
- `缺席-未执行`：方向重要，但缺少行动证据，且有拖延、替代投入或计划落空证据。
- `缺席-未记录`：用户做了，但日志没有记录过程、数量或结果。
- `目标变化`：用户主动调整了优先级。
- `证据不足`：无法判断，需要下周期补记录。

缺席项本身不是失败，而是校准信号。它提醒用户确认：这是目标偏离、记录失真，还是计划已经变化。

### 1. 回顾目标

可扩展内容：

- 当前方向锚点
- 原始目标
- 实际追求的目标
- 目标漂移
- 目标合理性
- 与上周期假说的关系

### 2. 评估结果

可扩展内容：

- 客观结果
- 主观感受
- 他人反馈
- 达成度判断
- 与预期的偏差
- 方向锚点完成度 / 缺席项检查

### 3. 分析原因（正向）

可扩展内容：

- 超预期的地方
- 关键成功链条
- 被忽略的优势
- 哪些做法值得放大

### 4. 分析原因（负向）

可扩展内容：

- 不符合预期的地方
- 根因分析
- 自我评价偏差
- 反例
- 数据边界 / 证据不足

### 5. 重来演练

可扩展内容：

- 如果重来一次会保留什么
- 会提前调整什么
- 最小替代动作是什么
- 哪些错误其实可以更早暴露

### 6. 后续规划

可扩展内容：

- 下个周期目标
- 手段路径
- 检查点
- 验证假说
- 风险提醒
- 重要缺席项的硬约束或补记录要求

## 四、回应复盘：从回应 AI 到校准方向

用户自己写复盘的意义，不是重复 AI 的周报 / 月报，而是完成方向确认、取舍和下注。

推荐在阅读 AI 复盘后写一份轻量回应复盘：

```markdown
## 0. 我的方向锚点

## 1. AI 复盘中我认可的判断

## 2. AI 复盘中我不认可或觉得偏的判断

## 3. AI 没提，但我认为重要的事

## 4. 本周期最大偏离风险

## 5. 下周期不可丢的硬约束
```

这份回应复盘的职责是：

- 确认哪些 AI 洞察进入生活。
- 指出 AI 没看见但用户认为重要的主线。
- 判断本周期是否偏离方向锚点。
- 把下周期最小行动约束写清楚。

一句话原则：

**AI 复盘事实和模式，用户复盘方向和取舍。**

## 五、三类复盘如何统一又区分

### 周复盘

- 仍然使用六问
- 每问更轻，只保留 1-2 个重点
- 更强调短周期调整
- 原因分析一般做到 `3 Why` 即可

### 月复盘

- 仍然使用六问
- 内部允许主题归并、多视角综合、上月假说验证
- 更强调模式识别、目标校准和下月路径

### 项目复盘

- 仍然使用六问
- 更强调里程碑、协作、流程、决策和复用经验
- 不只分析结果，还分析“机制为什么会产生这个结果”

## 六、项目复盘专用模板 / 命令骨架

项目复盘不是把月复盘换个名字，而是在同一个六问外壳下，换成更适合项目场景的内层槽位。

### 1. 回顾目标

- 这个项目 / 这轮优化原本想解决什么问题
- 验收标准是什么
- 过程中目标是否发生漂移

### 2. 评估结果

- 实际交付了什么
- 哪些目标达成，哪些未达成
- 当前是否通过验收，依据是什么

### 3. 分析原因（正向）

- 哪些动作最有效
- 哪些决策是对的
- 哪些协作方式、流程或机制值得复用

### 4. 分析原因（负向）

- 哪些地方仍不理想
- 更像是执行偏差、判断偏差、协作偏差，还是机制问题
- 哪些结论目前证据不足，只能暂定

### 5. 重来演练

- 如果重做一次，最应该提前做什么
- 哪些错误本可以更早暴露
- 最小但关键的替代动作是什么

### 6. 后续规划

- 下一轮要保留什么
- 要删除或替换什么
- 后续如何验证这次复盘结论是否成立

对应命令骨架：

```bash
/project-review 项目主题
/project-review 项目主题 --full
```

命令语义：

- `/project-review`：项目复盘专用入口
- `项目主题`：本次复盘对象或验收主题
- `--full`：开启更完整的项目机制分析深度

统一原则仍然不变：

**六问负责结构统一，项目专用槽位负责场景适配。**

## 七、周复盘的简化使用法

周复盘仍旧使用六问，只是每一问更短：

1. 回顾目标：本周目标是否明确，是否发生漂移
2. 评估结果：做到什么程度，主观感受和客观结果是否一致
3. 分析原因（正向）：哪些地方超预期，哪些做法值得保留
4. 分析原因（负向）：哪些地方低于预期，原因是什么
5. 重来演练：如果重来一周，最该提前做什么调整
6. 后续规划：下周目标、手段和检查方式

允许简化，但不建议删除“重来演练”。

## 八、常见误区

### 1. 把六问当成僵化问卷

错。

六问是导航骨架，不是限制内容深度的上限。你完全可以在六问内部做很强的主题综合。

### 2. 为了更高级的分析，删掉六问标题

错。

这样会破坏用户阅读稳定性，也会让不同复盘之间越来越难统一。

### 3. 把“重来演练”并入“后续规划”

不推荐。

它们很像，但职责不同：

- 重来演练：假设这次重做，哪里该改
- 后续规划：下一个周期怎么走

前者偏复盘修正，后者偏未来行动。

### 4. 只有六问，没有证据约束

也不够。

高质量复盘必须有证据、边界和反例，不能只靠主观感觉把六问填满。

## 九、推荐的一句话准则

如果只记一句：

**六问负责统一结构，主题综合负责提升质量。**
