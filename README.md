# 知己用户版

> 把日志贴进来，我来帮你看你自己没发现的模式。

这是 **知己** 的用户版。它保留了用户真正会用到的分析和复盘功能，不包含开发治理文档、内部计划和个人真实数据。

## 这个项目能做什么

- 分析单篇日志，给你一个更清晰的洞察和一个能马上尝试的动作
- 汇总一周或一个月的日志，生成周复盘、月复盘
- 对一个项目阶段做复盘
- 对长期方向做一次轻量的人生设计校准

## 怎么拿到项目

你可以用两种方式拿到这个用户版：

1. 向维护者索取用户版仓库地址，然后使用 Git 拉取
2. 直接拿压缩包，解压后进入项目目录

如果你是通过 Git 获取，常见方式如下：

```bash
git clone <维护者提供的地址>
cd zhiji-user
```

如果你会长期在这个仓库里写自己的真实日志和画像，建议额外执行一次：

```bash
powershell -ExecutionPolicy Bypass -File scripts/protect-private-data.ps1
```

它会保护 `关于我/` 下那些仍然需要保留模板入口、但你后续可能会写成私有内容的已跟踪文件，避免误提交。

## 第一次使用怎么开始

1. 把本项目放到 Claude Code / Codex 可以打开的目录中。
2. 进入本项目目录。
3. 直接粘贴一篇日志，然后输入：

```bash
/review
```

如果你只想明确测试单日反馈，也可以输入：

```bash
/daily-review today
```

## 没有自己的日志怎么办

可以直接用示例材料：

- 单日快速体验：[`examples/demo/sample-journal.md`](examples/demo/sample-journal.md)
- 一次覆盖周 / 月 / 项目 / 人生设计：[`examples/beta/README.md`](examples/beta/README.md)

如果你是第一次试用，优先沿用示例里的日志字段格式，比如“开心的事情 / 充实的事情 / 思考 / 待改进 / ToDolist”。

## 常用入口

| 入口 | 适合什么时候用 |
|------|----------------|
| `/review` | 不确定该用哪个命令时，优先用它 |
| `/daily-review` | 只分析某一天的日志 |
| `/weekly-review` | 已经有几天日志，想做周复盘 |
| `/monthly-review` | 已经有一整月日志，想做月复盘 |
| `/project-review` | 想对一个项目阶段、版本或验收过程做复盘 |
| `/life-design` | 感觉长期方向卡住了，想做一次校准 |
| `/journal-coach` | 想看最近几天的日志写作质量和改进点 |

## 输出会写到哪里

你使用过程中，项目会逐步在这些目录里产出内容：

- `日志/`：你的日志
- `复盘/每日反馈/`：日反馈和日志教练报告
- `复盘/每周复盘/`：周复盘
- `复盘/每月复盘/`：月复盘
- `复盘/项目复盘/`：项目复盘
- `复盘/人生设计/`：人生设计结果
- `关于我/`：长期上下文和验证沉淀

如果你是通过 Git 持续优化本项目，这些目录里的真实日志、复盘和画像类私有内容默认会被 Git 忽略，仓库里保留的是可分发模板。

## 如果你想逐项手动确认功能

可以直接照着这份清单执行：

- [`docs/user-smoke-test-checklist.md`](docs/user-smoke-test-checklist.md)

这份清单是用户视角的“人工烟测”说明，适合内测时快速判断：

- 命令能不能触发
- 会不会报缺文件 / 缺路径
- 结果有没有真的写到正确目录
- 返回内容是不是和命令类型一致

## 内测时最有价值的反馈

不需要写正式报告，只要告诉维护者这些就很有用：

- 哪个命令能跑通，哪个不能
- 报错原文是什么
- 哪一步最让你犹豫
- 哪些建议对你有用，哪些没用
- 后续日志里有没有回应前一次建议
