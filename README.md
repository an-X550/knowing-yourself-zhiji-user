# 知己用户版

> 把日志贴进来，用中文直接说你想做什么，它就会开始帮你分析。

这是 **知己** 的用户版。它的目标很简单：

- 帮你看清一篇日志里你自己没看到的模式
- 帮你把几天、几周、一个月的记录整理成复盘
- 帮你把“最近到底出了什么问题”说清楚
- 帮你从日志出发，慢慢形成更稳定的自我认知

它保留了用户真正会用到的分析和复盘功能，不包含开发治理文档、内部计划和个人真实数据。

## 先说最重要的：怎么直接开始

如果你使用 `WorkBuddy`、`Trae`、`Claude Code`、`Codex` 这类能直接打开项目目录并对话的工具，**打开本仓库后，直接粘贴日志就可以开始**。

推荐你优先用自然语言，不需要先理解命令：

```text
请分析今天这篇日志，告诉我：
1. 我没看到的模式
2. 最关键的问题
3. 明天最值得尝试的一个动作

（然后把日志正文贴上来）
```

如果你更习惯命令，也可以输入：

```bash
/review
```

可以把它理解成一句普通话：

```text
帮我分析这篇日志
```

对普通用户来说，**自然语言是推荐入口**，命令只是更短的快捷方式。

## 为什么这样设计

这个项目是从第一性原理出发设计的：

- 用户真正想要的不是“学会一套命令”，而是“把日志贴进去，然后得到有用反馈”
- 非编程用户最大的阻碍通常不是分析本身，而是“不知道该输入什么”
- 所以这里优先保留“直接粘贴内容 + 用中文描述需求”的体验
- 命令入口仍然保留，是为了高频用户、维护者和内测时快速复现

如果你主要使用国内生态，像 `WorkBuddy`、`Trae` 这样的工具会更贴近中文用户的日常使用方式；本项目也正是按“中文对话即可使用”的方向来组织入口的。

## 这个项目能做什么

### 1. 单篇日志分析

适合你刚写完一天日志，想快速看清今天发生了什么。

你会得到的通常是：

- 这篇日志里反复出现的模式
- 当前最值得关注的问题
- 一个可以马上尝试的小动作

推荐说法：

```text
分析这篇日志
```

或：

```bash
/review
```

### 2. 单日反馈

适合你已经把今天的日志放进项目里，想生成更标准的日反馈文件。

推荐说法：

```text
请帮我做今天这篇日志的反馈
```

或：

```bash
/daily-review today
```

### 3. 周复盘

适合你已经有至少几天日志，想知道这一周的状态、趋势、问题和进展。

推荐说法：

```text
请基于最近几天的日志帮我做周复盘
```

或：

```bash
/weekly-review
```

### 4. 月复盘

适合你已经积累了一个月左右的日志，想看更稳定的节奏、情绪、压力来源和成长轨迹。

推荐说法：

```text
请基于这个月的日志帮我做月复盘
```

或：

```bash
/monthly-review
```

### 5. 项目复盘

适合你刚做完一个版本、阶段、验收节点，想复盘“这个项目到底哪里做对了，哪里做偏了”。

推荐说法：

```text
请帮我复盘这个项目阶段
```

或：

```bash
/project-review
```

### 6. 人生设计校准

适合你不是单纯想优化一天，而是已经开始怀疑长期方向、目标和生活结构是否出了偏差。

推荐说法：

```text
我想重新校准一下最近的长期方向
```

或：

```bash
/life-design --quick
```

### 7. 日志教练

适合你想提升“写日志这件事本身”的质量，而不只是看日志内容。

推荐说法：

```text
请看看我最近几天的日志写作质量，告诉我怎么改进
```

或：

```bash
/journal-coach
```

## 第一次使用的最短路径

1. 用 `WorkBuddy`、`Trae`、`Claude Code` 或 `Codex` 打开本项目目录。
2. 打开一篇你自己的日志，或者先使用示例日志。
3. 把日志内容粘贴到对话里。
4. 直接输入一句中文，例如 `分析这篇日志`。
5. 如果你更习惯快捷方式，再使用 `/review`、`/weekly-review` 这类命令。

如果你只想最快体验一轮，直接走这条路径：

1. 打开 [`examples/demo/sample-journal.md`](examples/demo/sample-journal.md)
2. 复制其中任意一天
3. 粘贴到对话
4. 输入 `分析这篇日志`

## 没有自己的日志怎么办

可以直接用示例材料：

- 单日快速体验：[`examples/demo/sample-journal.md`](examples/demo/sample-journal.md)
- 一次覆盖周 / 月 / 项目 / 人生设计：[`examples/beta/README.md`](examples/beta/README.md)

如果你是第一次试用，优先沿用示例里的日志字段格式，比如“开心的事情 / 充实的事情 / 思考 / 待改进 / ToDolist”，这样更容易稳定触发正确分析路径。

## 命令入口和普通说法对照

| 命令入口 | 普通用户可以直接这样说 |
|------|----------------|
| `/review` | 帮我分析这篇日志 |
| `/daily-review` | 请帮我做今天这篇日志的反馈 |
| `/weekly-review` | 请基于最近几天日志帮我做周复盘 |
| `/monthly-review` | 请基于这个月日志帮我做月复盘 |
| `/project-review` | 请帮我复盘这个项目阶段 |
| `/life-design` | 我想重新校准一下最近的长期方向 |
| `/journal-coach` | 请看看我最近几天的日志写作质量 |

如果你不知道该说什么，就先用这一句：

```text
分析这篇日志
```

## 输出会写到哪里

你使用过程中，项目会逐步在这些目录里产出内容：

- `日志/`：你的原始日志
- `复盘/每日反馈/`：单日反馈和日志教练报告
- `复盘/每周复盘/`：周复盘
- `复盘/每月复盘/`：月复盘
- `复盘/项目复盘/`：项目复盘
- `复盘/人生设计/`：人生设计结果
- `关于我/`：长期上下文和验证沉淀

如果你是通过 Git 持续优化本项目，这些目录里的真实日志、复盘和画像类私有内容默认会被 Git 忽略，仓库里保留的是可分发模板。

如果你会长期在这个仓库里写自己的真实日志和画像，建议额外执行一次：

```powershell
powershell -ExecutionPolicy Bypass -File scripts/protect-private-data.ps1
```

它会保护 `关于我/` 下那些仍然需要保留模板入口、但你后续可能会写成私有内容的已跟踪文件，避免误提交。

## 内测时怎么判断它有没有正常工作

如果你想逐项手动确认功能，可以直接照着这份清单执行：

- [`docs/user-smoke-test-checklist.md`](docs/user-smoke-test-checklist.md)

这份清单适合内测时快速判断：

- 功能能不能触发
- 会不会报缺文件 / 缺路径
- 结果有没有真的写到正确目录
- 返回内容是不是和目标任务一致

## 内测时最有价值的反馈

不需要写正式报告，只要告诉维护者这些就很有用：

- 哪个入口能跑通，哪个不能
- 你是用自然语言触发的，还是用命令触发的
- 报错原文是什么
- 哪一步最让你犹豫
- 哪些建议对你有用，哪些没用
- 后续日志里有没有回应前一次建议

## 如果你是维护者

本仓库由上级项目 `knowing-yourself` 的导出流程刷新。共享逻辑（如 `.claude/`、`examples/`、`perspectives/` 与公共方法文档）应优先回补上级项目，再重新导出到这里；只属于分发体验的内容再直接维护在用户版覆盖层中。

刷新当前用户版时，请在上级仓库根目录执行：

```powershell
powershell -ExecutionPolicy Bypass -File scripts/export-zhiji-user.ps1
```

本用户版不包含开发规范、CHANGELOG、项目状态、内部计划和个人数据。
