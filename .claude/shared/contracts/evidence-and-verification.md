# 证据与验证契约

## 证据规则

1. 重要结论必须有日志日期、原文引用、已有报告或视角分析作为证据。
2. 没有证据时标注“证据不足”，不要补全、猜测或编造。
3. 发现要写成可被用户反驳的观察，不写成权威诊断。
4. 建议必须具体到行为和检查方式，避免“注意作息”“继续努力”“加强学习”等空泛表达。
5. 自检必须诚实。证据不足、数据不足、无昨日反馈时，用警告或 N/A 标记，不强行标满。

## 验证对象

验证时必须区分：

- **行为假说**：某种情境与行为、状态或结果之间可能存在的关系。
- **干预实验**：闹钟、午休、发邀请等具体方法在本次情境中是否有效。

一次干预失败通常只能说明“本次未奏效”，不能直接证伪它试图影响的行为假说。

## 验证沉淀文件

`context.verified_patterns` 是“预测 -> 实验 -> 验证”循环的沉淀文件。日反馈调用方负责写回，`daily-analyzer` 只负责返回证据化判断。

## 日反馈写回规则

1. **写回时机**：`/daily-review` 和 `log` skill 在保存 `output.daily_feedback` 后，读取 `context.verified_patterns` 并更新。
2. **写回来源**：以上一条日反馈的 `💊 新认知`、`⚡ 明天试试` 和本次 `⏮️` 验证判断为依据。
3. **状态词**：写回只使用 `待验证 / 部分支持 / 本次未奏效 / 多次支持 / 出现反例 / 证据不足`。
4. **状态规则**：
   - `✅ 做到了`：第一次只记“部分支持”；重复成立可记“多次支持”。
   - 行为假说至少 3 次独立支持、覆盖至少 2 个日期或情境且没有强反例，才能移入“已确认的模式”。
   - `❌ 没做`：具体行动记“本次未奏效”；同类行动连续 3 次没做时降低行动门槛或停止重复建议，不把行动失败直接写成行为假说被证伪。
   - `⚠️ 证据不足`：保留为“证据不足”，不升级、不证伪。
   - 明确反例先记“出现反例”；只有确定性预测被直接推翻，或多次独立反例稳定推翻同一假说时，才移入“已证伪的假说”。
5. **不硬凑成功**：没有日志证据时必须写“证据不足”，不能为了让 `verified-patterns.md` 变好看而判定成立。不得从“下一篇日志没有写”推导行动没有执行。

## 周/月/项目复盘消费

1. 周报和月报读取 `context.verified_patterns`，优先总结哪些假说得到部分 / 多次支持、出现反例、干预本次未奏效、连续没做或仍待验证。
2. 如果 `context.verified_patterns` 为空或未更新，报告中标注“验证沉淀缺失”，并把下一周期第一优先级设为补齐日反馈验证闭环。
3. 复盘不应优先增加新行动数量；当待验证假说堆积或行动连续没做时，下周期规划必须先降低行动门槛并提高验证密度。
4. 对连续没做的行动，降低门槛或停止重复建议；不要把同一无效建议包装成新建议。
